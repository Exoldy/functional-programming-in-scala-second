`choiceN`, список там пиздец использовался как функция типа `Int => Par[A]`! Давай замутим более общую сигнатуру, чтоб все эти случаи в неё влезли, как матрёшки одна в другую:

![Страница 0199, изображение 1](./assets/page-0199/image-01.png)

```scala
extension [A](pa: Par[A])
  def chooser[B](choices: A => Par[B]): Par[B]
```

#### УПРАЖНЕНИЕ 7.13

Замути этот новый примитив `chooser`, а потом на его костях собери `choice` и `choiceN`.

Как только обобщаешь такие функции, всегда кидай на них критический взгляд, будто на код-ревью у старпёра. Функция могла родиться из конкретного кейса, но сигнатура и имплементация вдруг оказываются универсальнее, чем ты думал. Здесь `chooser` уже не самый охуенный нейм для этой хуйни — она реально общая: параллельная вычислуха, которая при запуске сначала катит первую тачку, а её результат определяет, что катить во второй. Вторая вычислуха даже не обязана существовать заранее, до результата первой. Не надо её пихать в контейнер, как в `List` или `Map`. Может, она генерится на лету из первого результата, из ничего, как кролик из шляпы. Эта функция везде в FP-библиотеках торчит, обычно зовётся `bind` или `flatMap` (flatMap):

```scala
extension [A](pa: Par[A])
  def flatMap[B](f: A => Par[B]): Par[B]
```

`flatMap` — это реально самый примитив, или можно ещё обобщить? Давай поиграемся, как в песочнице с типами. Имя `flatMap` намекает, что операцию можно разложить на два шага: *маппинг* (`map`) `f: A => Par[B]` по нашему `Par[A]`, который генерит `Par[Par[B]]`, а потом *флэттенинг* (`flatten`) этого вложенного `Par[Par[B]]` в плоский `Par[B]`. Но это ж интересно, блядь; выходит, хватило добавить ещё проще комбинатор — назовём его `join` — для конверсии `Par[Par[X]]` в `Par[X]` при любом `X`:

```scala
def join[A](ppa: Par[Par[A]]): Par[A]
```

Опять же, просто следуем типам, как по стрелкам в IntelliJ. Есть пример, который требует функцию с такой сигнатурой — ну и создаём её из воздуха. Теперь, когда она есть, можно подумать, что эта сигнатура значит на самом деле. Зовём её `join`, потому что концептуально это параллельная вычислуха, которая при запуске выполнит внутреннюю, дождётся её (как `Thread.join`, только без драматизма), и вернёт результат.
