> Часть 2: Функциональный дизайн и библиотеки комбинаторов / Глава 9: Комбинаторы парсеров / 9.5 Отчёт об ошибках / 9.5.2 Вложенность ошибок

Давай поэтапно разберём наши комбинаторы для обработки ошибок — как на код-ревью, когда стек-трейс (stack trace) молчит, а ты чуешь подвох. Начнём с очевидного, как дважды два. Ни один из примитивов пока не позволяет прилепить к парсеру конкретное сообщение об ошибке. Вводим базовый комбинатор для этого — `label`:

```scala
extension [A](p: Parser[A])
  def label(msg: String): Parser[A]
```

Идея `label` в том, что если `p` обосрётся, то его `ParseError` как-то впитает в себя `msg`. А что под этим подразумевается на деле? Можем, конечно, тупо предположить `type ParseError = String` и что возвращаемый `ParseError` будет равен лейблу, но нам-то хочется, чтоб ошибка ещё и место косяка указывала — как нормальный логгер в продакшене, а не мемный "ошибка где-то здесь". Давай пока добавим это в нашу алгебру:

```scala
case class Location(input: String, offset: Int = 0):
  lazy val line = input.slice(0, offset + 1).count(_ == '\n') + 1
  lazy val col = input.slice(0, offset + 1).lastIndexOf('\n') match
    case -1         => offset + 1
    case lineStart  => offset - lineStart
```

![Страница 0259, изображение 1](./assets/page-0259/image-01.png)

> Вытаскивает реальную локацию из абстрактной `ParseError` — чтоб знать, на какой строке/столбце парсер решил сдаться.

![Страница 0259, изображение 2](./assets/page-0259/image-02.png)

```scala
def errorLocation(e: ParseError): Location
def errorMessage(e: ParseError): String
```

> Вытаскивает реальное сообщение из абстрактной `ParseError` — без этой воды про "неожиданный конец".

Мы тут выбрали конкретное воплощение для `Location`: полный инпут, оффсет в него, плюс номер строки и колонки — последнее лениво вычисляется из инпута и оффсета, чтоб не жрать память зря. Теперь можем точнее сказать, чего ждём от `label`. При фейле с `Left(e)`, `errorMessage(e)` будет ровно тем сообщением, что задал `label`. Это можно чётко прописать через `Prop`:

```scala
def labelLaw[A](p: Parser[A], inputs: SGen[String]): Prop =
  forAll(inputs ** Gen.string):
    case (input, msg) =>
      p.label(msg).run(input) match
        case Left(e) => errorMessage(e) == msg
        case _       => true
```

А что насчёт `Location`? Хотим, чтоб реализация `Parsers` сама её заполняла локацией, где парсер накосячил. Пока это всё ещё мутновато, как неопределённость в квантовой физике: если у нас `a | b`, и оба парсера фейлятся на инпуте, то какая локация и лейбл(ы) вылезут в отчёте? Разберём в следующей секции, не ссыте.

### 9.5.2 Вложенность ошибок

Хватит ли комбинатора `label` для всех нужд по отчётам об ошибках? Пока нет, пацаны, не всё так просто — я сам через это прошёл, когда парсеры жрали недели. Взглянем на пример:

```scala
val p = string("abra").label("first magic word") **
  string(" ").many **
  string("cadabra").label("second magic word")
```

> Скипаем whitespace (пробелы, табы и прочий воздух).
