## 205 8.6 Ответы на упражнения

Можем заимплементить `||` похожим манером — сначала жмём левую проперти, а правую трогаем только если левая наебнулась, как в жизни, блядь:

```scala
extension (self: Prop) def ||(that: Prop): Prop =
(n, rng) => self(n, rng) match
case Falsified(_, _) => that(n, rng)
case x => x
```

Давай ещё зафигачим способ запускать наши проперти, чтоб в REPL потыкать и поэкспериментировать (позже в главе разжуём эту тему с проверкой и запуском пропертей; пока просто хотим вызвать underlying функцию и результат на консоль вывалить, без изысков):

```scala
extension (self: Prop)
def run(): Unit =
self(100, RNG.Simple(System.currentTimeMillis)) match
case Falsified(msg, n) =>
println(s"! Falsified after $n passed tests:\n $msg")
case Passed =>
println(s"+ OK, passed $testCases tests.")
```

Метод `run()` прогоняет проперти по 100 тест-кейсам с `RNG`, заточенной под текущее время. Потом результаты в консоль херачит. Имплементы `&&` и `||` рабочие, но когда фейл — инфы как кот наплакал, хуй поймёшь, где собака зарыта. Взгляни на примеры: `p` всегда проходит как по маслу, а `q` рандомно сливает, как flaky тест в CI:

```scala
val p: Prop = Prop.forAll(Gen.boolean)(x => x == x)
val q: Prop = Prop.forAll(Gen.boolean)(x => x)
```

Когда гоняем композитные проперти, слепленные из `&&` и `||`, нихуя не ясно, какая именно дочерняя проперти подставила:

```scala
scala> (p && q).run()
! Falsified after 4 passed tests:
false
scala> (q && p).run()
! Falsified after 1 passed tests:
false
scala> (q || q).run()
! Falsified after 2 passed tests:
false
```
