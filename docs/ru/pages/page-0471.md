> Часть 4: Эффекты и I/O / Глава 15: Обработка потоков и инкрементальный I/O / 15.4 Применения

Можно ли впихнуть эти возможности в нашу дефиницию `Stream` так, чтоб безопасность ресурсов не просралась? Конечно, хуйня вопрос, можно! У нас уже вся мощь наготове, через комбинатор `flatMap`, который мы сами слепили для произвольного типа `Stream`. Например, `flatMap` плюс наши текущие комбинаторы позволяют первый сценарий отыметь вот так:

```scala
def convertAll(inputFile: String, outputFile: String): Task[Unit] =
lines(inputFile)
.flatMap(lines).pipe(conversion).pipe(writeLines(outputFile)).run
```

Этот код полностью безопасен по ресурсам — все файловые хэндлы runner сам закроет, как только закончит с ними дело, даже если эксепшены прилетят нахуй. Писать в несколько файлов? Просто меняем порядок вызовов `flatMap`, и поехали:

```scala
def convertMultisink(inputFile: String): Task[Unit] =
lines(inputFile).flatMap(file =>
lines(file).pipe(conversion).pipe(writeLines(file + ".celsius"))).run
```

### 15.4 Применения

Идеи из этой главы — это как швейцарский нож в FP, везде применимо. Охуенно много программ можно перековать в stream processing — как только эта абстракция тебе в мозг въестся, она везде из всех щелей торчит, как тараканы в общаге. Давайте глянем на домены, где это реально тащит:

- **Ввод-вывод с файлами (File I/O)** — Мы уже на примере показали, как stream processing жрёт файловый ввод-вывод. Хотя в примерах ковырялись по строчкам, библиотека запросто тянет и бинарники, без вопросов.

- **Обработка сообщений, конечные автоматы и акторы (actors)** — Крупные системы часто лепят как кучу слабо связанных компонентов, которые треплются сообщениями. Это обычно выражают через *акторов (actors)*, с явными send/receive (отправка/получение). А мы можем такие компоненты описать как stream-вычисления — и вуаля, суперсложные автоматы и поведения собираются на высокоуровневом композиционном API, без гемора.

- **Серверы и веб-приложения** — Веб-апп можно видеть как преобразователь потока HTTP-запросов в поток HTTP-ответов. Просто и элегантно, как в жизни.

- **UI-программирование (user interface)** — Отдельные UI-события вроде кликов мышкой — это стримы, а весь UI — огромная сеть stream-вычислений, которая решает, как интерфейс дрочит на взаимодействия юзера.

- **Big data и распределённые системы** — Stream-библиотеки легко распределяются и параллелизуются для тонны данных. Ключевой инсайт: ноды в сети stream-processing не обязаны жить на одной тачке, они могут разлететься по кластеру, как косяк на фесте.
