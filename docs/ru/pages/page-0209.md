### 8.1 Краткий тур по property-based тестированию

Для примера возьми ScalaCheck ([ScalaCheck](http://mng.bz/n2j9)) — это библиотека для property-based тестов в Scala, где свойство выглядит примерно так, как в следующем листинге.  
Короче, не просто тесты на фиксированных примерах, а генератор хаоса, который сам ищет, где твой код обосрётся.

![Страница 0209, изображение 1](./assets/page-0209/image-01.png)

**Листинг 8.1. Свойства ScalaCheck**

*Генератор списков целых чисел от 0 до 100*

```scala
import org.scalacheck.{Gen, Prop}
val intList = Gen.listOf(Gen.choose(0, 100))
val prop =
  Prop.forAll(intList)(ns =>
    ns.reverse.reverse == ns) &&
  Prop.forAll(intList)(ns =>
    ns.headOption == ns.reverse.lastOption)
val failingProp =
```

*Свойство, которое описывает поведение метода `List.reverse`*

*Проверяем, что разворот списка дважды возвращает оригинал.*

```scala
Prop.forAll(intList)(ns => ns.reverse == ns)
```

*Проверяем, что первый элемент становится последним после разворота.  
Свойство, которое заведомо ложно — чисто для демо, чтоб увидеть, как оно наебнётся.*

И проверяем свойства вот так:

```scala
scala> prop.check
+ OK, passed 100 tests.
scala> failingProp.check
! Falsified after 6 passed tests.
> ARG_0: List(0, 1)
```

Тут `intList` — это не просто `List[Int]`, а `Gen[List[Int]]`, штука, которая знает, как нагенерить тестовые данные типа `List[Int]`.  
Сэмплишь из неё — и вуаля, списки разной длины, набитые рандомными числами от 0 до 100, как пьяный тестировщик на код-ревью шарит по edge-кейсам (краевым случаям).  

Генераторы в property-based либах — это вообще мощная хуйня с богатым API: комбинируй, компонуй, переиспользуй, как лего для тестов.  
Функция `Prop.forAll` клеит свойство из генератора типа `Gen[A]` и предиката `(A => Boolean)`.  
Свойство утверждает: все значения, которые нагенерит, должны проходить предикат, иначе — фейл.  

Рисунок 8.1 показывает, как `intList` связан с одним из свойств. Как генераторы, так и свойства — с кучей API.  
В простом примере мы `&&` склеили два свойства. Итоговое держится, только если ни одно не наебнулось на генерируемых кейсах.  
Вместе они дают частичную спецификацию верного поведения метода `reverse`<sup>1</sup> — не полная Библия, но уже не "works on my machine".

Когда зовёшь `prop.check`, ScalaCheck рандомно нагенерит `List[Int]` значений, чтоб найти кейс, где предикат обосрётся.  
Вывод говорит: сгенерено 100 тестовых кейсов (типа `List[Int]`), и все прошли.<sup>1</sup>

<sup>1</sup> Цель такого тестирования — не расписать поведение по пикселям, а просто поднять уверенность в коде, чтоб не словить продакшен-пиздец.  
Как с обычными тестами: можно свойства углублять до бесконечности, но всегда взвешивай cost–benefit, стоит ли оно того, или лучше пивка допить.
