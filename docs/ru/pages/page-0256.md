После этого раздела вернёмся к вопросу, как впихнуть нормальную отчётку об ошибках в наш парсерный API. Сделаем это аккуратно, не ломая общую архитектуру и почти не трогая наш JSON-парсер. Плюс, слепим реальную, запускаемую инкарнацию нашего типа `Parser`. Ключевой момент: парсер JSON, который накатаем в следующем разделе, будет работать независимо от этой реализации, как партизан в тылу врага.

### 9.4.1 Формат JSON

Если вы ещё не в теме всех деталей формата JSON, зацените спецификацию грамматики (http://json.org). Вот пример JSON-документа:

```scala
{
  "Company name" : "Microsoft Corporation",
  "Ticker" : "MSFT",
  "Active" : true,
  "Price" : 30.66,
  "Shares outstanding" : 8.38e9,
  "Related companies" : [
    "HPQ", "IBM", "YHOO", "DELL", "GOOG"
  ]
}
```

*Значение* (*value*) в JSON может быть одним из нескольких типов. *Объект* (*object*) — это последовательность пар ключ-значение, разделённых запятыми и обёрнутых в фигурные скобки `{ }`. Ключи должны быть строками вроде `"Ticker"` или `"Price"`, а значения — это либо другой объект, либо *массив* (*array*), типа `["HPQ", "IBM" … ]`, содержащий другие значения, либо *литерал* (*literal*), вроде `"MSFT"`, `true`, `null` или `30.66`. Напишем довольно дебильного парсера, который просто выжмет синтаксическое дерево из документа, не заморачиваясь дальнейшей обработкой.<sup>11</sup> Нам понадобится представление для распарсенного JSON-документа. Введём для этого тип данных:

```scala
enum JSON:
  case JNull
  case JNumber(get: Double)
  case JString(get: String)
  case JBool(get: Boolean)
  case JArray(get: IndexedSeq[JSON])
  case JObject(get: Map[String, JSON])
```

### 9.4.2 Парсер JSON

Вспомним, что мы накатали такой набор примитивов:

- `string(s)` — распознаёт и возвращает один `String`
- `regex(s)` — распознаёт регулярное выражение `s`
- `p.slice` — возвращает кусок инпута, осмотренный `p`, если успешно

<sup>11</sup>Смотрите заметки к главе (https://github.com/fpinscala/fpinscala/wiki) для обсуждения альтернативных подходов.
