### 11.5.2 Монда State и частичное применение типов

Вспомни, как мы в шестой главе ковыряли тип данных `State`. Мы там напилили комбинаторы для `State`, включая `map` и `flatMap` — те самые, что цепляют состояние, как крюки на скалодроме.

**Листинг 11.7. Возвращаемся к нашему типу `State`**

```scala
opaque type State[S, +A] = S => (A, S)
object State:
extension [S, A](underlying: State[S, A])
def map[B](f: A => B): State[S, B] =
flatMap(a => unit(f(a)))
def flatMap[B](f: A => State[S, B]): State[S, B] =
s =>
val (a, s1) = underlying(s)
f(a)(s1)
```

Смотрите, `State` идеально ложится под шаблон монды, как перчатка на руку. Но вот засада: её конструктор типов жрёт два аргумента, а `Monad` требует ровно один, так что просто так не скажешь `Monad[State]`. Зато если зафиксировать какой-нибудь конкретный `S`, то получается `State[S, *]` — и это уже то, что ждёт `Monad`. Короче, у `State` не одна монда-инстанция, а целая семья — по одной на каждый выбор `S`. Хотим частично применить `State`, зафиксировав `S` на бетоне, — это как частичное применение функции, только на уровне типов, без этой imperative хуйни. Например, лепим конструктор `IntState` — алиас для `State` с первым аргументом прибитым гвоздями к `Int`:

```scala
type IntState[A] = State[Int, A]
```

А `IntState` — это как раз то, под что можно слепить `Monad`, без вопросов:

```scala
given intStateMonad: Monad[IntState] with
def unit[A](a: => A): IntState[A] = State(s => (a, s))
def flatMap[A,B](st: IntState[A])(f: A => IntState[B]): IntState[B] =
State.flatMap(st)(f)
```
