> Часть 1: Введение в функциональное программирование / Глава 4: Обработка ошибок без исключений / 4.6 Ответы на упражнения

- Тип `Try` — это как `Either`, только ошибки не какая-то хуйня произвольного типа, а аккуратно в `Throwable`. Ограничив их подтипами `Throwable`, `Try` раздаёт кучу удобных методов для кода, который привык кидать эксепшены (exceptions) направо-налево, как в imperative-кошмаре (imperative nightmare).

- Тип `Validated` — брат-близнец `Either`, но когда комбайнишь несколько фейлов, ошибки не улетают в жопу, а накапливаются, чтоб потом разом разобрать этот бардак.

- Функции высшего порядка вроде `map` и `flatMap` — это как спасательный круг: ковыряешься в потенциально сломанных вычислениях, не матча ошибку на каждом сраном вызове. Такие HOF'ы (higher-order functions) висят у всех этих типов обработки ошибок, чтоб не изобретать велосипед заново.

### 4.6 Ответы на упражнения

#### ОТВЕТ 4.1

```scala
enum Option[+A]:
case Some(get: A)
case None
def map[B](f: A => B): Option[B] = this match
case None => None
case Some(a) => Some(f(a))
def getOrElse[B>:A](default: => B): B = this match
case None => default
case Some(a) => a
def flatMap[B](f: A => Option[B]): Option[B] =
map(f).getOrElse(None)
def orElse[B>:A](ob: => Option[B]): Option[B] =
map(Some(_)).getOrElse(ob)
def filter(f: A => Boolean): Option[A] =
flatMap(a => if f(a) then Some(a) else None)
```

Методы `map` и `getOrElse` пилим паттерн-матчингом (pattern matching) по `this`, с чётким поведением для каждого конструктора данных — никаких сюрпризов, пацаны. `flatMap` и `orElse` реализуем через вложенные опшены (Option), как матрёшку, а потом `getOrElse` сдираем внешний слой. Наконец, `filter` лепим через `flatMap`, точь-в-точь как для `List` — копипастим проверенную логику, без самодеятельности.

#### ОТВЕТ 4.2

```scala
def variance(xs: Seq[Double]): Option[Double] =
mean(xs).flatMap(m => mean(xs.map(x => math.pow(x - m, 2))))
```
