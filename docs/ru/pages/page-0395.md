> Часть 4: Эффекты и I/O / Глава 13: Внешние эффекты и I/O / 13.3 Избегание переполнения стека (StackOverflowError) / 13.3.2 Траполин (trampoline): Общее решение проблемы переполнения стека

```scala
FlatMap(a1, a1 =>
  FlatMap(a2, a2 =>
    FlatMap(a3, a3 =>
      ...
      FlatMap(aN, aN => Return(aN)))))
```

Если теперь закинуть `unsafeRun` на нашу тестовую прогу `p`, она будет вертеться вечно без переполнения стека — именно то, что доктор прописал.  
Наша `unsafeRun` не даст стеку лопнуть даже на бесконечно рекурсивных `IO`-монстрах.  

Что мы тут наворотили, пацаны? Когда JVM-машина зовёт функцию, она как пьяный на код-ревью кидает фрейм в колл-стек, чтоб потом вернуться и допить кофе.  
Мы этот цирк с контролем потока вытащили на свет в нашем `IO`-типе.  

`unsafeRun`, интерпретируя `IO`-прогу, чует: то ли эффект через `Suspend(s)` запускать, то ли подпрограмму через `FlatMap(x, f)` дёргать.  
Вместо того чтоб стек раздувать как воздушный шарик на оптимизации хвостовых вызовов (tail-call optimization, TCO), которая в Scala иногда подводит, `unsafeRun` сам вызывает `x()`, потом `f` на результате, а `f` мгновенно кидает `Suspend`, `FlatMap` или `Return` — и контроль снова у `unsafeRun`, как в кооперативной многозадачности (cooperative multitasking) из 90-х.  

Наш `IO` — чистой воды корутина (*coroutine*), который дружит с `unsafeRun` как старые кореша: шлёт запросы `Suspend` или `FlatMap`, приостанавливается, отдаёт руль, а `unsafeRun` толкает вперёд по одному саспенду, как трамплин (*trampoline*) для рекурсивного прыжка.  
Функция вроде `unsafeRun` — это трамплин (*trampoline*), а вся херня называется трамполинингом (*trampolining*) — возвращаешь контроль в один большой цикл, и стек остаётся чистым, как совесть после рефакторинга.

### 13.3.2 Траполин (trampoline): Общее решение проблемы переполнения стека

В нашем `IO`-монаде `resume`-функции вообще не обязаны ебаться с сайд-эффектами (side-effects) — это не требование.  
Этот `IO` — универсальный траполин (*trampoline*) для любых вычислений, даже чистых, где I/O и не пахнет!  

`StackOverflowError` (переполнение стека) в Scala выскакивает везде, где композированная функция накидала вызовов больше, чем стек выдержит — классика, через которую каждый FP-ветеран прошёл, как через чистилище рекурсивных парсеров в проде.  
Демонстрировать проще простого:

> Мы строим композированную функцию `g` из 100 000 функций, где каждая вызывает следующую.

```scala
scala> val f = (x: Int) => x
val f: Int => Int = <function1>
scala> val g = List.fill(100000)(f).foldLeft(f)(_ compose _)
val g: Int => Int = <function1>
scala> g(42)
java.lang.StackOverflowError
```

Она, бля, упадёт и на куда меньших композициях — попробуй сам, если не веришь, стек JVM не резиновый.  
К счастью, фиксим нашим `IO`-монадом на раз-два:

```scala
scala> val f = (x: Int) => IO.Return(x)
val f: Int => IO[Int] = <function1>
```
