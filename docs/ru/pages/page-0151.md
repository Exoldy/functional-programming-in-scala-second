### 6.3 Делаем stateful API чистыми

Эта засада — как превратить кажущиеся stateful API в чистые, и её разбор  
(API просто вычисляет следующий стейт, а не мутирует хрень на месте) —  
не только у генерации рандома. Она вылазит постоянно, как тараканы в общаге,  
и всегда разруливается одним и тем же способом.[^3]

Допустим, у тебя есть класс такой:

```scala
class Foo:
  private var s: FooState = ...
  def bar: Bar
  def baz: Int
```

Предположим, `bar` и `baz` каждый по-своему ебут `s`.  
Мы можем механически перевести эту хрень в чистый FP API,  
сделав переход от стейта к стейту явным, как в хорошем контракте:

```scala
trait Foo:
  def bar: (Bar, Foo)
  def baz: (Int, Foo)
```

Как только юзаем этот паттерн, коллер сам тащит вычисленный next state  
по всему коду, как чемодан без колёс.  
Для чистого интерфейса `RNG` (RNG, Random Number Generator),  
который только что показали, если переиспользуешь старый `RNG`,  
он всегда высрёт то же самое значение. Например:

```scala
def randomPair(rng: RNG): (Int, Int) =
  val (i1, _) = rng.nextInt
  val (i2, _) = rng.nextInt
  (i1, i2)
```

![Страница 0151, изображение 1](./assets/page-0151/image-01.png)

Здесь `i1` и `i2` будут одинаковые, как близнецы из матрицы!  
Чтобы выдать два разных числа, нужно взять `RNG`, который вернула первая `nextInt`,  
и им генерить вторую `Int`:

> Возвращаем финальный стейт после двух рандомов.  
> Это позволяет коллеру продолжить генерить новые значения с этим стейтом.

```scala
def randomPair(rng: RNG): ((Int, Int), RNG) =
  val (i1, rng2) = rng.nextInt
  val (i2, rng3) = rng2.nextInt
```

> Обрати внимание на юз `rng2` тут.

```scala
((i1, i2), rng3)
```

Видишь общий паттерн, а?  
И как это может заебать напрямую юзать этот API,  
как копать лопатой асфальт.  
Давай напишем пару функций для генерации рандомов  
и глянем, нет ли повторухи, которую можно вынести в DRY (DRY, Don’t Repeat Yourself).

[^3]: Потеря в эффективности от вычисления next state чистыми функциями —  
потому что нельзя мутировать данные на месте, как в imperative (императивном) дерьме.  
(Тут не проблема, стейт — один `Long`, скопировать раз плюнуть.)  
Эту потерю можно сгладить эффективными чисто функциональными структурами данных.  
Иногда можно и на месте помутить, не ломая референциальную прозрачность (referential transparency),  
— об этом в части 4.
