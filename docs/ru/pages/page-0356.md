## 327 12.5 Законы аппликативных функторов

Закон ассоциативности для аппликативов — та же хуйня в общем-то. 
Без него пришлось бы плодить по две версии `map3` — типа `map3L` и `map3R` — 
в зависимости от скобок, и привет, взрыв остальных комбинаторов, 
потому что все группировки разные, как в том аду с парсерами. 
Формулируем ассоциативность через `product`[^6]. 
Вспомним, `product` просто склеивает два эффекта в пару через `map2`:

```scala
extension [A](fa: F[A])
  def product[B](fb: F[B]): F[(A, B)] =
    fa.map2(fb)((_, _))
```

Если пары вложены справа, всегда можно перекинуть их в левые — без вопросов:

```scala
def assoc[A, B, C](p: (A, (B, C))): ((A, B), C) =
  p match
    case (a, (b, c)) => ((a, b), c)
```

С этими `product` и `assoc` закон ассоциативности аппликативов выглядит вот так:

```scala
fa.product(fb).product(fc) == fa.product(fb.product(fc)).map(assoc)
```

Обратите внимание, вызовы `product` слева ассоциированы влево, справа — вправо от знака `==`. 
А справа потом маппим `assoc`, чтоб финальные тюплы легли ровно, как в тетрисе.

### 12.5.3 Натуральность произведения

Последний закон аппликативов — *натуральность*, чтоб всё коммутало как надо. 
Покажем на простом примере с `Option`, без лишней воды.

**Листинг 12.6.** Получение имён сотрудников и годовой зарплаты

```scala
case class Employee(name: String, id: Int)
case class Pay(rate: Double, hoursPerYear: Double)
def format(employee: Option[Employee], pay: Option[Pay]): Option[String] =
  employee.map2(pay)((e, p) =>
    s"${e.name} makes ${p.rate * p.hoursPerYear}"
  )
val employee: Option[Employee] = ...
val pay: Option[Pay] = ...
format(employee, pay)
```

[^6]: `product`, `map` и `unit` — альтернативная формулировка `Applicative` (Applicative).
