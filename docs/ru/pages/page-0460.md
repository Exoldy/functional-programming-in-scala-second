## 431 15.3 Расширяемые pull'ы и стримы

```scala
val dequeue: IO[Message] = ...
val logAll: Stream[IO, Unit] =
  Stream.eval(dequeue).repeat.map(format).pipe(log)
```

В этом примере `dequeue` тупо блочит, пока сообщение не прилетит (ну, скажем, из распределённой очереди сообщений, типа Kafka или Rabbit). 

`logAll` в бесконечном цикле дёргает `dequeue`, каждое сообщение форматирует в строку и валит в консоль логом. 

Общий стрим — это `Stream[IO, Unit]`. 

Как эту хрень запустить? У нас только два элиминатора: `fold` и `toList`. 

Если `toList` — полный пиздец, каждый лог выдаст Unit, они накопятся в билдере списка, как снежный ком, и сожрут всю кучу памяти под чистую. 

Вместо этого фолдим и каждый Unit в помойку кидаем. 

Это классика с эффектными стримами — вечно вылазит в проде, так что давай новый элиминатор замутим:

```scala
extension [F[_], O](self: Stream[F, O])
  def run(using Monad[F]): F[Unit] =
    fold(())((_, _) => ()).map(_(1))

val program: IO[Unit] = logAll.run
```

### 15.3.2 Обработка ошибок

Некоторые эффектные типы дают возможность ловить ошибки, которые вылазят при вычислении. 

В коде главы 13 `Task[A]` — это opaque-обёртка (opaque wrapper) вокруг `IO[Try[A]]`. 

`Task` позволяет игнорить потенциальные траблы, пока не решишь ими заняться — как отложить бомбу под сиденье до финального босса:

```scala
val a: Task[Int] = Task("asdf".toInt)
val b: Task[Int] = a.map(_ + 1)
val c: Task[Try[Int]] = b.attempt
val d: Try[Int] = c.unsafeRunSync(pool)
```

Когда таск `a` отрабатывает, конверт строки в инт валится с `NumberFormatException` (NumberFormatException). 

`Task` это ловит, запихивает в `Failure` внутри (помнишь, `Failure` — это конструктор `Try`?). 

В определении `b` `map` превращается в no-op (no-op), короткое замыкание. 

В `c` `attempt` вытаскивает подложку `Try[Int]`. 

`Task` добавляет к `IO` *канал ошибок* — как запасной сток для дерьма, чтоб ошибки текли сквозь компьютацию, точно так же, как `Try` и `Either` для чистых вычислений без эффектов. 

Можем `Monad[Task]` слепить, как для `Try` и `Either` — с short-circuiting (short-circuiting) на `Failure`, чтоб не плодить лишние вычисления. 

`Task` ещё даёт `handleErrorWith`, выведенный из `attempt` и `flatMap`, — классический паттерн для "поймал ошибку — подмени на другую":

```scala
def handleErrorWith(h: Throwable => Task[A]): Task[A] =
  attempt.flatMap:
    case Failure(t) => h(t)
    case Success(a) => Task(a)
```
