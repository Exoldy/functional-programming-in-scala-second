# Страница 0201

> Часть 2: Функциональный дизайн и библиотеки комбинаторов / Глава 7: Чисто функциональный параллелизм / 7.6 Ответы на упражнения

Мы моделируем параллельные вычисления как значения типа данных `Par`, а отдельный интерпретатор `run` уже сам спавнит эти треды и гоняет их в реале. 
В следующей главе нырнём в совсем другой домен, опять поплывём зигзагами к API для него и выжмем ещё порцию уроков по функциональному дизайну — как в том меме, где кот пытается собрать IKEA, но в итоге строит космический корабль.

### Итоги

- Ни одна готовая либа не святая корова — всегда можно разобрать по винтикам. В большинстве библиотек куча произвольных решений на отъебись. Накидать альтернативу — и бац, домен раскрывается с новых сторон, как пасхалка в старом Doom.

- Простые примеры рубят прямиком к сути домена, не давая утонуть в случайном шуме и boilerplate-говне (boilerplate).

- Параллельные и асинхронные вычисления можно загнать в чисто функциональные рамки, без мутабельного ада.

- Строим описание вычисления плюс отдельный интерпретатор, который его крутит, — и вуаля, вычисления сами становятся значениями, их можно комбайнить с другими, как Lego.

- Крутой приём для API-дизайна: conjureишь типы и импы (implementations), пытаешься слепить, корректируешь, итератируешь — чистый TDD (test-driven development) на стероидах, прошёл через это в проде, знаю, где собака зарыта.

- Тип `Par[A]` описывает вычисление, которое может разогнать часть или всё по нескольким тредам — как оркестр, где дирижёр не дерётся с музыкантами.

- Значения `Par` трансформируешь и комбайнишь знакомыми операциями вроде `map`, `flatMap` и `sequence` — никаких сюрпризов, всё как в FP-мечтах (FP — functional programming).

- API как алгебра с законами, которые прижимают импы к стенке, — это и дизайн-инструмент, и тестовая фича в одном флаконе, самоирония: сам когда-то законы забил, и привет, баги в проде.

- Разбиваем API на минимальный набор примитивов и комбинаторов — переиспользование прёт, понимание растёт, как экспонента в алгоритмах.

- Актор — это неблокирующий (non-blocking) примитив параллелизма (concurrency) на сообщениях, как почтальон в многопоточке. Акторы сами по себе не чисто функциональны, но ими реализуешь FP-API на ура, как мы показали имплементацией `map2` для неблокирующего `Par` — пост-ирония в чистом виде.

### 7.6 Ответы на упражнения

#### ОТВЕТ 7.1

Давай закинем параметр типа и для параллельных вычислений, и для выхода той функции, которая комбайнит их результаты — как апгрейд, чтоб не жёстко типизировать под каждый случай:

```scala
def map2[A, B, C](pa: Par[A], pb: Par[B])(f: (A, B) => C): Par[C]
```
