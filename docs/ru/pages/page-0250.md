## 221 9.2 Возможная алгебра

У нас жёсткие ожидания от `map`: он должен просто перелопатить результат, если `Parser` отработал успешно. Ни хуя лишнего из инпута ковырять не надо, и мёртвый парсер через `map` не воскреснет, а живой не сдохнет. Короче, `map` — *структуру сохраняющий*, как мы от `Par` и `Gen` требовали, помните, пацаны? Давайте это знакомым законом формализуем:

```scala
p.map(a => a) == p
```

Как этот закон задокументировать? Можно в док-коммент запихнуть, но в прошлом коде мы наворотили способ делать законы *исполняемыми*, как живые тесты. Используем эту хуйню здесь.

**Листинг 9.2.** Комбинируем `Parser` с `map`

```scala
import fpinscala.testing.*

trait Parsers[ParseError, Parser[+_]]:
  ...
  object Laws:
    def equal[A](p1: Parser[A], p2: Parser[A])(in: Gen[String]): Prop =
      Prop.forAll(in)(s => p1.run(s) == p2.run(s))
    def mapLaw[A](p: Parser[A])(in: Gen[String]): Prop =
      equal(p, p.map(a => a))(in)
```

Это потом выстрелит, когда будем проверять, что наша `Parsers`-имплементация не тупит. Когда ещё законов накопаем, смело пиши их проперти в `Laws` объекте.[^8]

Кстати, раз `map` у нас в кармане, `char` можно на `string` набить, без лишнего геморроя:

```scala
def char(c: Char): Parser[Char] =
  string(c.toString).map(_.charAt(0))
```

По аналогии, комбинатор `succeed` — на `string` с `map`:

```scala
def succeed[A](a: A): Parser[A] =
  string("").map(_ => a)
```

Этот парсер всегда выдаст `a`, и похуй на инпут (ведь `string("")` прокатит везде, даже на пустом). Знакомый комбинатор, а? Его поведение законом опишем:

```scala
succeed(a).run(s) == Right(a)
```

[^8]: Ещё раз: смотри код главы за примерами. Чтоб главу не растягивать, не лепим `Prop` для всех законов, но это не отмазка — сам пиши!
