## 213. Ответы на упражнения 8.6

- Если юзаем `hashCode`, чтоб дёргать стейт `RNG`, то генерируемая функция высасывает **всю** инфу из `String`, чтоб влиять на генерируемое значение `A`. Как голодный вампир, жрёт контекст целиком. Только входные строки с **одинаковым** `hashCode` гарантированно родят **тот же самый** `A` — никаких сюрпризов, как в продакшене после деплоя.

- А если хапаем `length`, то функция ковыряет лишь **кусочки** инфы из `String`, чтоб слепить `A`. Для всех строк одной длины — бац, **гарантированно** одинаковый `A`. Как ленивый мем с котом: "длина совпадает — и похуй на остальное".

Стратегия зависит от того, какие функции для тестов реалистичны, пацаны. Хотим монстров, что жрут весь вход целиком для результата, или скромнее — те, что тыкают в биты и байты по вкусу? Обернем эту херню в тип, чтоб не путаться:

```scala
opaque type Cogen[-A] = (A, RNG) => RNG
```

С этим новым типом обобщаем `genStringFn` под любой входной тип — универсал, как `map` на стероидах:

```scala
def fn[A, B](in: Cogen[A], out: Gen[B]): Gen[A => B] =
State[RNG, A => B]: rng =>
val (seed, rng2) = rng.nextInt
val f = (a: A) => out.run(in(a, rng2))(0)
(f, rng2)
```

Но вот засада: как юзеру показать фейл теста? При обломе он увидит только "непрозрачная (opaque) функция сломала свойство" — ну охуеть информативно, как лог "ошибка произошла". В Haskell-овском QuickCheck (http://mng.bz/MvYW) уже ковыряли это: репортят юзеру и shrink'ят функции до минимального вида, который всё равно опровергает (falsifies) свойство. Типа, "вот эта хуйня минимально сломана". Зацените talk Малкольма Уоллеса про shrinking и showing функций с Haskell Symposium 2012: http://mng.bz/aZq7... Я сам через такое прошёл — когда shrink не сработал, пришлось вручную дебажить, матерясь в три ручья.
