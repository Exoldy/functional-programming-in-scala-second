## 185 8.1 Краткий тур по тестированию на основе свойств

В этой схеме `Prop` — чистый аналог4 ленивого `Boolean`, и все стандартные булевы функции (AND, OR, NOT, XOR и прочая херня) можно на него навешать — но голого `Boolean` явно мало, пацаны. Когда свойство накрывается пиздецом, охота знать, сколько тестов оно сожрало до этого, какие аргументы его добили, а если выжило — сколько раундов оно перетерпело. Давай попробуем отдавать `Either` на успех или фейл, как нормальные люди:

![Страница 0214, изображение 1](./assets/page-0214/image-01.png)

> Такие opaque-алиасы типов реально качают читаемость и типобезопасность API, чтоб не путаться в этой хуйне.

```scala
object Prop:
opaque type SuccessCount = Int
...
trait Prop:
def check: Either[???, SuccessCount]
```

Какой тип пихать в фейл-кейс? Мы нихуя не знаем о типах генерируемых тестовых кейсов — они как чёрные ящики из ада. Добавить параметр типа в `Prop` и слепить `Prop[A]`? Тогда `check` мог бы плеваться `Either[A,Int]`. Но стопэ, прежде чем нырять в эту кроличью нору, спросим себя по-честному: а нахуя нам тип значения, которое свойство угробило? Нам похуй на него; тип нужен только если дальше с этим фейлом ебаться в вычислениях. А мы скорее всего просто на экран его сплюнем, чтоб тестировщик разобрался. В конце концов, цель — баги выковыривать и показывать, какие кейсы их триггерят, чтоб фиксили, а не в типах копаться. Правило железное: `String` не для данных, с которыми считаем, — это для лохов. Но для показа людям — в самый раз, как мемасик в чате. Так что отмазываться можно такой представкой `Prop`:

```scala
object Prop:
opaque type FailedCase = String
opaque type SuccessCount = Int
trait Prop:
def check: Either[(FailedCase, SuccessCount), SuccessCount]
```

При фейле `check` кидает `Left((s, n))`, где `s` — строка с тем самым говном, что свойство угандошило, а `n` — сколько кейсов прошло до катастрофы. Это фиксит возврат `check` — ну, на данный момент. А что с аргументами `check`? Пока их ноль, как в пустом пиве. Хватит ли? Давай глянем, к чему `Prop` имеет доступ, просто поковыряв, как они рождаются. Особенно `forAll` — вот где засада:

```scala
def forAll[A](a: Gen[A])(f: A => Boolean): Prop
```

4 Эквивалентно в том смысле, что `Prop` можно слепить как `() => Boolean` вместо трейта с одной хуйней-методом — выразительность не просрётся.
