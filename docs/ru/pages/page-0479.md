#### ОТВЕТ 15.12

Сначала разберёмся, пацаны, как `attempt` ебётся с теми конструкторами, которые мы можем потрогать (ну, типа `unit` и `raiseError`, как в том меме про "доступные интерфейсы, а под капотом пиздец"):

```scala
unit(a).attempt == unit(Right(a))
raiseError(t).attempt == unit(Left(t))
```

Теперь глянем, как `handleErrorWith` с каждым из них дерётся по полной — без пощады, как на код-ревью у старшего:

```scala
unit(a).handleErrorWith(h) == unit(a)
raiseError(t).handleErrorWith(h) == h(t)
```

А вот так `raiseError` с `flatMap` общаются — чисто как два алкаша в баре, один другому в рожу плюёт:

```scala
raiseError(t).flatMap(f) == raiseError(t)
```

Можем ещё жёстче требования накидать на эксепшены, чтоб не как в том продакшене, где всё летит к чертям. Например, должен ли `unit` ловить те бомбы, что thunk ему подсовывает, или пусть самообезвоживается?

```scala
unit(throw new Exception) == raiseError(new Exception)
```

А если функции, кинутые в `map` и `flatMap`, взорвутся эксепшенами — хватать их за яйца или пускать на самотёк, как в legacy-коде?

```scala
fa.map(_ => throw new Exception) == raiseError(new Exception)
fa.flatMap(_ => throw new Exception) == raiseError(new Exception)
```

Правильного ответа нет, только трейд-оффы — выбирай, где тебе меньше крови из ушей пойдёт, я сам через это говно 16 лет ковырялся.
