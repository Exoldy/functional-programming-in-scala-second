## 251 9.8 Ответы на упражнения

```scala
case Success(a, n) =>
buf += a
go(p, offset + n)
case f @ Failure(e, true) => f
case Failure(e, _) => Success(buf.toList, offset)
case Slice(n) =>
buf += s.input.substring(offset, offset + n)
go(p, offset + n)
go(p, 0)
```

Тут пиздец как много наворочено, пацаны! Если состояние орёт, что мы в слайсе, то запускаем рекурсивную функцию — парсит до первого косяка, но главное — **не тащим** результаты каждой итерации за собой, как алкаш тележку с бутылками!

Наткнулись на uncommitted failure (незакреплённую неудачу) — бац, возвращаем `Slice` с счётчиком сожранных символов. Каст выглядит как тёмный переулок ночью, стрёмно до усрачки, но chill, он safe as fuck: мы ж знаем, результат слайсованный, так что выходной `List[A]` никто и не потрогает, мем про "не трогай руками" в чистом виде.

Если слайса нет — та же рекурсия, но теперь сваливаем все спарсенные вкусняшки в буфер, как в корзину на Авито. Остальные комбинаторы тоже в теме слайсов быть обязаны (типа `map2` и `flatMap`, чтоб не облажаться). Полный разбор в гите, не ленитесь ковыряться: http://mng.bz/vovM.

#### ОТВЕТ 9.18

Давай перелопатим `ParseError`, чтоб он хранил **все** ошибки, которые вылезли при парсинге, а не только главную:

```scala
case class ParseError(stack: List[(Location,String)] = Nil,
otherFailures: List[ParseError] = Nil):
def addFailure(e: ParseError): ParseError =
copy(otherFailures = e :: otherFailures)
```

Тогда `or` реализуем так, чтоб трекал косяки из **обеих** веток, без потерь:

```scala
extension [A](p: Parser[A])
infix def or(p2: => Parser[A]): Parser[A] =
s => p(s) match
case Failure(e, false) => p2(s).mapError(_.addFailure(e))
case r => r
```

Но эта херня, как снежный ком, тянет за собой доработки. Придётся апгрейдить `toString` в `ParseError`, чтоб он эти фейлы рендерил по-человечески — полезно и интуитивно, а не стеной текста как в старом логе. Плюс комбинаторы завести для чистки парс-ошибок: обрезать хлам, фильтровать до самых жирных, чтоб не тонуть в дерьме, как в типичном прод-логе.
