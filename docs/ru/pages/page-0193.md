Функция `A => Unit` иногда зовётся *continuation* (продолжение) или *callback* (колбэк) — ну, вы поняли, классика жанра, как колбэк из ада в JS, только здесь не так больно. С таким кодированием, когда мы применяем `ExecutorService` к функции, представляющей `Par[A]`, на выходе получаем новую функцию: `(A => Unit) => Unit`. Её потом можно вызвать, сунув колбэк, который обработает выданный `A`. Этот колбэк сработает ровно тогда, когда `A` наконец-то посчитается — не сразу, а когда звёзды сойдутся.

Локальные сайд-эффекты для чистого API. Тип `Future`, который мы тут наваяли, выглядит довольно императивно — `A => Unit`, бля, прям как `void main()` из C. Такая функция годится только на то, чтоб какой-то сайд-эффект на `A` отжечь, потому что возвращаемый результат мы явно в жопу не используем. А мы всё ещё пишем функциональное программирование с таким `Future`? Конечно, хуй там плавал! Просто юзаем классический трюк: сайд-эффекты как внутренняя хуйня для чистого функционального API. Это прокатывает, потому что эти побочки не видны коду, который `Par` юзает. Функциональное представление `Future` — opaque (непрозрачное), как чёрный ящик, снаружи его не вызовешь, не подглядишь.

Пока мы допиливаем остаток неблокирующей `Par`, убеди себя, что эти сайд-эффекты не просочатся наружу — чисто для паранойи. Идеи локальных эффектов, наблюдаемости и тонкостей чистоты с референциальной прозрачностью (referential transparency) мы разберём в главе 14 по полной, а пока интуитивно хватит, не ссы.

С таким представлением `Par` давай сначала реализуем `run` — изменим её, чтоб она просто `A` возвращала. Поскольку она идёт от `Par[A]` к `A`, придётся continuation слепить и ему `Future` скормить.

Листинг 7.6 Реализация `run` для `Par`

![Страница 0193, изображение 2](./assets/page-0193/image-02.png)

> Мутабельная, потокобезопасная ссылка для хранения результата (загляни в `java.util.concurrent.atomic` за инфой про эти классы)

```scala
extension [A](pa: Par[A]) def run(es: ExecutorService): A =
val ref = new AtomicReference[A]
val latch = new CountDownLatch(1)
pa(es) { a => ref.set(a); latch.countDown }
latch.await
ref.get
```

> Когда значение прилетает, оно сетит результат и дропает защёлку.

> `java.util.concurrent.CountDownLatch` позволяет потокам ждать, пока `countDown` не вызовут нужное число раз. Тут `countDown` сработает один раз, когда `A` от `p` прилетит, и `run` будет блочить, пока это не случится.

> Ждёт, пока результат не появится и защёлка не слетит.

> Как только защёлку прошли, знаем, что `ref` сетанута, и возвращаем её значение.
