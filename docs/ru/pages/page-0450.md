## 421 15.2 Простые трансформации потоков

```scala
def fold[A](init: A)(f: (A, O) => A): A =
self.fold(init)(f)(1)
def toList: List[O] =
self.toList
def take(n: Int): Stream[O] =
self.take(n).void
def ++(that: => Stream[O]): Stream[O] =
self >> that
given Monad[Stream] with
def unit[A](a: => A): Stream[A] = Pull.Output(a)
extension [A](sa: Stream[A])
def flatMap[B](f: A => Stream[B]): Stream[B] =

```

```scala
object Pull:
extension [O](self: Pull[O, Unit])
```

> Конвертирует `Pull[O, Unit]` в `Stream[O]`

```scala
def toStream: Stream[O] = self
```

Кроме непрозрачного определения типа и инстанса монады,  
мы накидали ещё пару дополнительных API.  
Методы `toPull` и `toStream` позволяют прыгать туда-сюда между `Stream` и `Pull`,  
как по bunny-hop'у в Counter-Strike.  

Мы также подогнали API для прямой работы со стримами —  
делегируем под капотом на соответствующие операции `Pull`,  
но подкручиваем тип возвращаемого значения, где без этого никуда.  
Взять хотя бы `take` на `Stream`: возвращает `Stream[O]`,  
а `Option[R]` от `take` на `Pull` просто в помойку —  
классика, когда удобство бьёт по финтифлюшкам.  

Ещё один забавный прикол — операция `++`,  
которая склеивает два стрима через монадическую последовательность подложенных pull'ей,  
как будто лепишь франкенштейна из двух половинок.  

А нахуя вообще оба `Pull` и `Stream`?  
`Pull` явно мощнее (смотри, `Stream` фиксит результат в `Unit`, как нищий в подвале),  
так почему не сесть на `Pull` верхом и не забыть про этот проблемный монадный инстанс,  
который только нервы треплет?  

`Stream` чаще удобнее в деле — моделирует стриминговые вычисления  
как коллекцию наподобие `LazyList`, лениво жрёт данные, не торопясь,  
а `Pull` — это король для ковыряния кастомных стриминговых компьютаций  
на `uncons` и `flatMap`, где `flatMap` пляшет как в трансе.  

Мы опираемся на эти фичи при дизайне API:  
выoulouровневые, коллекционные комбинаторы лепим прямо в `Stream`,  
чтоб пацаны не мучились,  
а операции для сборки новых стриминговых монстров — в `Pull`,  
чтоб не размазывать по стенке сложностью.

### 15.2.2 Композиция трансформаций потоков

Тип `Pull` позволяет описывать любые, мать их, сложные источники данных  
через монадическую рекурсию — как будто рисуешь бесконечный лабиринт из pull'ей.  
А `Stream` подкидывает API в стиле коллекций,  
которое ковыряет по одному элементу, вываливающемуся из pull'а,  
как из рога изобилия.  

Все стриминговые трансформации, что мы напилили до этого,  
висят методами на `Pull` или `Stream`.  
Но нам охота определять трансформации стримов независимо от самого стрима  
и компоновать несколько в один жирный комбайн.  
И это решается классической композицией функций — просто, элегантно, без лишнего говна!

```scala
type Pipe[-I, +O] = Stream[I] => Stream[O]
```
