# Страница 0289

> Часть 3: Общие структуры в функциональном дизайне / Глава 10: Моноиды / 10.3 Ассоциативность и параллелизм

Левосторонний фолд выглядел бы вот так:

```scala
combine(combine(combine(a, b), c), d)
```

А сбалансированный фолд — это уже другая песня, смотри:

```scala
combine(combine(a, b), combine(c, d))
```

Балансированный фолд позволяет параллелизм, потому что два внутренних `combine` полностью независимы — запускай их одновременно, и не парься. Но это ещё цветочки: дерево по-балансированнее работает эффективнее, когда каждый `combine` жрёт время пропорционально размеру аргументов. Классика жанра — глянь на runtime этой хуйни:

```scala
List("lorem", "ipsum", "dolor", "sit").foldLeft("")(_ + _)
```

На каждом шаге фолда аллоцируем полную промежуточную `String`, чтобы сразу её выкинуть и слепить ещё жирнее в следующем. Помнишь, `String` — неизменяемая (immutable), и `a + b` — это свежая куча символов, куда копируем оба по полной, время O(`a.length + b.length`). Вот трассировка этого цирка:

```scala
List("lorem", "ipsum", "dolor", "sit").foldLeft("")(_ + _)
List("ipsum", "dolor", "sit").foldLeft("lorem")(_ + _)
List("dolor", "sit").foldLeft("loremipsum")(_ + _)
List("sit").foldLeft("loremipsumdolor")(_ + _)
List().foldLeft("loremipsumdolorsit")(_ + _)
"loremipsumdolorsit"
```

Видишь, промежуточные строки плодятся как кролики, а потом дохнут пачками. Нормальный подход — комбайнить по половинкам, это и зовётся балансированным фолдом: сначала `"loremipsum"` и `"dolorsit"`, потом их в один кулак моноидом.

#### УПРАЖНЕНИЕ 10.7

Затяви `foldMap` для `IndexedSeq`.^4 Имплементация должна резать последовательность пополам, рекурсивно прогонять каждую половину и комбайнить результаты моноидом:

```scala
def foldMapV[A,B](as: IndexedSeq[A], m: Monoid[B])(f: A => B): B
```

^4 Напомню, `IndexedSeq` — интерфейс для неизменяемых (immutable) структур с быстрым случайным доступом по индексу. Плюс у него эффективные `splitAt` (разделить по позиции) и `length` (длина).
