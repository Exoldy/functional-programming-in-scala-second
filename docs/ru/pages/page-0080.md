## 51 3.4 Trees

Алгебраические типы данных — это как универсальный конструктор для твоих структур, пацаны. Давай слепим простую бинарную деревошную хрень, чтоб не ебаться с массивами:

```scala
enum Tree[+A]:
case Leaf(value: A)
case Branch(left: Tree[A], right: Tree[A])
```

Паттерн-матчинь снова выручает, как старый кореш на код-ревью, — позволяет удобно ковыряться в элементах нашего ADT. Давай напишем функцию `size`, которая посчитает все ноды (листья с ветками) в дереве, чтоб не гадать на кофейной гуще:

![Страница 0080, изображение 2](./assets/page-0080/image-02.png)

```scala
enum Tree[+A]:
case Leaf(value: A)
case Branch(left: Tree[A], right: Tree[A])
```

> Паттерн-матчинь сюда, чтоб понять, на Leaf или Branch наш size вызывают — классика, через которую все мы прошли, как через первый merge conflict.

```scala
def size: Int = this match
case Leaf(_) => 1
case Branch(l, r) => 1 + l.size + r.size
```

> Рекурсивно дёргает size на каждой ветке — рекурсия, блядь, как бесконечный скролл в твиттере, но с выходом.

Операцию `size` мы засунули как метод прямо в `Tree` — чтоб вызывать в ооп-стиле, как нормальные люди, а не как хипстеры с их функциями. Это упрощает и определение, и юзанье, без лишнего геморроя. Взгляните, как бы выглядела standalone-функция `size`:

```scala
object Tree:
def size[A](t: Tree[A]): Int = t match
case Leaf(_) => 1
case Branch(l, r) => 1 + size(l) + size(r)
```

Два определения — как близнецы, мать их! Синтаксис методов даёт проще вызовы (`t.size` вместо `size(t)`) и чище сигнатуры. Когда лепишь дата-структуры, выбирай: методы или функции — поэкспериментируй, сам поймёшь, где подвох, как я в 2008-м на первом FP-проекте.
