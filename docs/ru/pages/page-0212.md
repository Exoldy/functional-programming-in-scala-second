## 183 8.1 Краткий тур по property-based тестированию

### 8.1.2 Первые наброски API

Ладно, с этим разобрались, поехали нахуй в бой. Какие дата-типы нам нужны для нашей тестовой либы? Какие примитивы определять, и что они вообще значат? Какие законы должны наши функции блюсти? Как и раньше, берём простой пример, вычитываем нужные типы и функции — и смотрим, что вылезет. Для вдохновения глянем на пример из ScalaCheck, который показывали ранее:

```scala
val intList = Gen.listOf(Gen.choose(0, 100))
val prop =
Prop.forAll(intList)(ns => ns.reverse.reverse == ns) &&
Prop.forAll(intList)(ns => ns.headOption == ns.reverse.lastOption)
```

Не зная ни хуя про реализацию `Gen.choose` или `Gen.listOf`, можно угадать, что тип, который они возвращают (назовём его `Gen` — сокращение от *generator*, короче), должен быть параметризован каким-то типом. То есть, `Gen.choose(0,` `100)` скорее всего возвращает `Gen[Int]`, а `Gen.listOf` — это функция с сигнатурой `Gen[Int]` `=>` `Gen[List[Int]]`. Но блядь, не похоже, чтобы `Gen.listOf` парился насчёт типа Gen, который получает на вход (было бы пиздец странно требовать отдельные комбинаторы для списков `Int`, `Double`, `String` и прочей хуйни), так что сделаем полиморфным, как нормальные пацаны:

```scala
def listOf[A](a: Gen[A]): Gen[List[A]]
```

Из этой сигнатуры можно выудить кучу инсайтов, будто из лога продакшена стек-трейс. Заметьте, чего мы *не* указываем — размер списка для генерации. Чтобы это было реализуемо, генератор либо сам предполагает размер, либо ему говорят. Предполагать — хуйня негибкая, как монолит на монолите, любое предположение не везде впишется. Значит, генераторам нужно *говорить* размер тест-кейсов. Представьте API, где это явно:

```scala
def listOfN[A](n: Int, a: Gen[A]): Gen[List[A]]
```

Это точно полезный комбинатор, но отсутствие явных размеров — это вообще ракета. Тест-раннер сам решает, какие размеры тест-кейсов юзать, и открывается дверь для той минимизации кейсов, о которой мы болтали раньше. Если размеры всегда жёстко зашиты программистом, то раннеру пиздец, никакой гибкости. Держите эту хуйню в уме, пока копаем дизайн глубже — я сам на этом спотыкался, когда первую пропертя-тестовую либу клеил, чуть не обосрался. А что с остальным примером? Функция `Prop.forAll` выглядит зачётно; видно, что она жрёт `Gen[List[Int]]` и соответствующий предикат: `List[Int]` `=>` `Boolean`. Но опять же, `forAll` не должен ебаться с типами генератора и предиката, лишь бы совпадали. Выразим это типом:

```scala
def forAll[A](a: Gen[A])(f: A => Boolean): Prop
```
