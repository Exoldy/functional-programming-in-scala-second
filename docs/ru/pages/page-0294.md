## 265 10.6 Foldable структуры данных

```scala
val intMultiplication: Monoid[Int] = new:
def combine(a1: Int, a2: Int) = a1 * a2
val empty = 1
given string: Monoid[String] with
def combine(a1: String, a2: String) = a1 + a2
val empty = ""
val booleanOr: Monoid[Boolean] = new:
def combine(x: Boolean, y: Boolean) = x || y
val empty = false
val booleanAnd: Monoid[Boolean] = new:
def combine(x: Boolean, y: Boolean) = x && y
val empty = true
```

Тут мы объявили ``intAddition`` как канонический ``Monoid[Int]``, а ``intMultiplication`` просто как обычное значение — чтоб не путаться, как в том старом меме про "default everything". Аналогично, канонический инстанс для ``Monoid[String]`` зафичили. А ``booleanOr`` и ``booleanAnd`` пустили как raw-значения, потому что дефолтные из них — полное дерьмо, не для продакшена. Когда зовёшь функцию с контекстными параметрами, можешь подсунуть свои инстансы вместо given'ов через ключевик ``using``. Например:

```scala
val allPositive = foldMap(List(1, 2, 3))(_ > 0)(using Monoid.booleanAnd)
val existsNegative = foldMap(List(1, 2, 3))(_ < 0)(using Monoid.booleanOr)
```

За деталями по given'ам (`https://mng.bz/Bx02`) и контекстным параметрам (`https://mng.bz/do7w`) лезьте в оффициалку — там не то что в наших старых Cats-доках, где полдня ковыряешься. Подведём итог нашей хуйне: у нас параметризованный трейт — ``Monoid[A]`` — с каноническими (дефолтными) инстансами для кучи типов. Плюс generic-функции, которые жрут инстансы трейта как контекст. Это и есть квинтэссенция *typeclass*-паттерна — трейт с одним тип-параметром и пачкой given-инстансов. Typeclass — это ad hoc полиморфизм в чистом виде: интерфейс лепишь независимо от типа, а потом типы под него подстраиваешь, как Lego в руках опытного девопса. Я через это прошёл на рефакторинге legacy-кода, когда visitor'ы из ООП чуть не угробили дедлайны. Остаток главы будем тащить ``Monoid`` как typeclass, а не как голое значение; монойды пихать в контекст, инстансы — как given'ы, без этой values-херни.
