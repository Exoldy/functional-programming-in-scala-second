## 335 12.7 Применения Traverse

### 12.7.3 Комбинирование траверсабельных структур

Природа траверса такая, что он заебенивает сохранять форму аргумента — это и его суперсила, и одновременно ахиллесова пята. Это наглядно вылазит, когда пытаешься слить две структуры в одну цельную хрень. Допустим, у тебя есть `Traverse[F]`, можно ли взять значение какого-то типа `F[A]` и второе того же сорта `F[B]`, и слепить из них `F[C]`? Ладно, давай попробуем `mapAccum` юзать, чтоб набросать generic-версию `zip`, как в старые добрые времена, когда всё казалось проще.

Listing 12.12 Комбинирование двух разных типов структур

```scala
extension [A](fa: F[A])
def zip[B](fb: F[B]): F[(A, B)] =
fa.mapAccum(fb.toList):
case (a, Nil) => sys.error("zip: Incompatible shapes.")
case (a, b :: bs) => ((a, b), bs)
.apply(0)
```

Эта версия `zip` тупо не тянет аргументы разной формы — классический подвох, через который я сам не раз спотыкался в проде. Например, если `F` это `List`, то она не справится со списками разной длины. В этой имплементации список `fb` должен быть не короче `fa`, иначе пиздец. А если `F` это `Tree`, то `fb` на каждом уровне должен иметь не меньше веток, чем `fa`. Можем слегка подкрутить эту generic `zip` и выдать две версии, где форма одной стороны доминирует, как альфа-самец в стае.

Listing 12.13 Более гибкая имплементация `zip`

```scala
extension [A](fa: F[A])
def zipL[B](fb: F[B]): F[(A, Option[B])] =
fa.mapAccum(fb.toList):
case (a, Nil) => ((a, None), Nil)
case (a, b :: bs) => ((a, Some(b): Option[B]), bs)
.apply(0)
def zipR[B](fb: F[B]): F[(Option[A], B)] =
fb.mapAccum(fa.toList):
case (b, Nil) => ((None, b), Nil)
case (b, a :: as) => ((Some(a): Option[A], b), as)
.apply(0)
```

Такие варианты заебись работают для `List` и других последовательностей — чистый кайф, без лишнего геморроя. В случае с `List`, например, результат `zipR` возьмёт форму от `fb` и подложит `None` слева, если `fb` длиннее `fa`, как заплатки на старые джинсы. А для типов посложнее, типа `Tree`, это может оказаться не тем, что нужно — привет, реальность FP, где не всё идеально симметрично. Обратите внимание, в `zipL` мы просто сплющиваем правый аргумент в `List[B]` и выкидываем его структуру в топку. Для `Tree` это выйдет в preorder-траверс лейблов по нодам, как DFS по древу без возврата. Потом зиппим эту последовательность лейблов с значениями левого `Tree,` `fa`, не пропуская несоответствующие поддеревья — полный фарш. Для деревьев `zipL` и `zipR` реально полезны, только если заранее знаешь, что оба дерева одной формы, иначе это как пытаться наложить разные маски на одно лицо в меме.
