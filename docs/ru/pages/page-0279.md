# Страница 0279

> Часть 2: Функциональный дизайн и библиотеки комбинаторов / Глава 9: Комбинаторы парсеров / 9.8 Ответы на упражнения

Тип `Slice` жрёт один параметр — сколько символов из инпута мы откусили, как мясник на конвейере.  
Заметим, что `Slice` extends (наследует от) `Result[String]`; это классический GADT (generalized algebraic data type, обобщённый алгебраический тип данных), где кейсы не просто варианты, а настоящие типовые снайперы, уточняющие параметры, чтоб Scala не дал тебе споткнуться о собственные ноги.  
В отличие от vanilla (обычного) ADT (алгебраического типа данных), тут один или несколько кейсов зажимают типовые параметры в тиски.  
Конкретно, `Slice` фиксит тип `A` как `String` — никаких компромиссов, чистая типовая диктатура.  

Когда мы паттерн-матчим по `Result[A]` и попадаем в кейс `Slice`, Scala внезапно становится телепатом: знает, что `A` — это `String`, и юзает эту инфу для дальнейшей типовой мясорубки.  
Представьте код-ревью, где компилятор заранее видит твой баг и орёт "не пущу!".  
Взять хотя бы этот override (переопределение) `map` для `Parser`:

```scala
override def map[B](f: A => B): Parser[B] =
  s => p(s) match
    case Success(a, n) => Success(f(a), n)
    case Slice(n) => Success(f(s.slice(n)), n)
    case f @ Failure(_, _) => f
```

В теле (body) кейса `Slice` Scala infers (выводит): наш парсер `p` — чистой воды `Parser[String]`, иначе парсинг не выдал бы нам `Slice`, как кот пустую миску.  
Следовательно, функция `f` — это стрелка из `String` прямиком в `B`, и Scala милостиво позволяет apply (применить) `s.slice(n)` (которая плюётся `String`) к `f`.  

С такими definitions (определениями) мы перелопачиваем комбинатор `slice`, чтоб он ставил флаг `isSliced` в состоянии парсера — как галочку "я не лузер":

```scala
extension [A](p: Parser[A])
  def slice: Parser[String] =
    s => p(s.copy(isSliced = true)) match
      case s @ Slice(_) => s
      case Success(_, length) => Slice(length)
      case f @ Failure(_, _) => f
```

Комбинатор `slice` ещё и результат трансформирует, гарантируя, что любой `Success` заменяется на `Slice` — никаких полумер, полная замена, как апгрейд с Java на Scala.  
Наконец, впихиваем `isSliced` во все комбинаторы, оптимизируя их под "ленивый режим": если `isSliced` равно `true`, то computation (вычисление) результата пропускаем, как ненужный мем на стриме.  
Вот как это выглядит для `many`:

```scala
override def many: Parser[List[A]] =
  s =>
    var nConsumed: Int = 0
    if s.isSliced then
      def go(p: Parser[String], offset: Int): Result[String] =
        p(s.advanceBy(offset)) match
          case f @ Failure(e, true) => f
          case Failure(e, _) => Slice(offset)
          case Slice(n) => go(p, offset + n)
          case Success(_, _) =>
            sys.error("sliced parser should not return success, only slice")
      go(p.slice, 0).asInstanceOf[Result[List[A]]]
    else
      val buf = new collection.mutable.ListBuffer[A]
      def go(p: Parser[A], offset: Int): Result[List[A]] =
        p(s.advanceBy(offset)) match
```
