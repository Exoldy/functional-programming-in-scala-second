Выглядит как-то криво, да? `None` теперь значит "всё заебись, тесты прошли, свойство держится", а `Some` — "пиздец, фейл". Раньше мы юзали только кейс `None` от `Option` чисто для фейлов, а тут он обозначает отсутствие оных. Это вполне легитимный трюк с `Option`, но интент размыт, как кофе без кофеина. Давай замутим свежий дата-тип, эквивалентный `Option[(FailedCase, SuccessCount)]`, чтоб всё на лицо было, без домыслов.

**Листинг 8.2. Создание дата-типа `Result`**

> Указывает, что один из тестовых кейсов фальсифицировал (falsified) свойство

```scala
enum Result:
case Passed
case Falsified(
failure: FailedCase, successes: SuccessCount)
```

> Указывает, что все тесты прошли rows × cols раз

```scala
def isFalsified: Boolean = this match
case Passed => false
case Falsified(_, _) => true
```

Теперь это нормальное представление для `Prop`? Давай ещё разок глянем на `forAll`. Можно ли его заимплементить? Почему да или нет?

```scala
def forAll[A](a: Gen[A])(f: A => Boolean): Prop
```

Видно невооружённым взглядом, что у `forAll` инфы кот наплакал, чтоб выдать `Prop`. Кроме числа тестов для прогона, запуск `Prop` должен тащить за собой всю кухню для генерации этих самых тестов. Если генерить рандомные кейсы на базе нашего текущего RNG (генератор случайных чисел), то без `Gen` никуда — нужен `RNG`. Давай протащим эту зависимость прямиком в `Prop`:

```scala
opaque type Prop = (TestCases, RNG) => Result
```

Если нарисуются другие зависимости — помимо числа тестов и источника рандома, — просто накидаем их доп. параметрами в `check` позже, без гемора. Теперь у нас хватает мяса, чтоб заимплементить `forAll`. Вот простая реализация в следующем листинге.

**Листинг 8.3. Имплементация `forAll`**

```scala
def forAll[A](as: Gen[A])(f: A => Boolean): Prop =
(n, rng) =>
randomLazyList(as)(rng)
.zip(LazyList.from(0))
.take(n)
.map:
case (a, i) =>
try
```
