#### УПРАЖНЕНИЕ 8.14

Забабахай свойство, которое проверит поведение `List.sorted` (загляни в доки API: http://mng.bz/N4En), им, кстати, можно сортировать (и не только) `List[Int]`.9 Например, `List(2,` `1,` `3).sorted` равно `List(1,` `2,` `3)`. ### 8.2.3 Пишет тестовый сьют для параллельных вычислений

Помнишь, в главе 7 мы откопали законы, которые должны держаться для наших параллельных расчётов, как скала в шторм? А ну-ка, выразим их нашей либой? Первое, что мы ковыряли, было конкретным тестовым кейсом:

```scala
unit(1).map(_ + 1) == unit(2)
```

Мы, блядь, конечно, можем выразить этот закон для `Par`, но выходит какая-то многоэтажная хуйня:10

```scala
val executor: ExecutorService = Executors.newCachedThreadPool
val p1 = Prop.forAll(Gen.unit(Par.unit(1)))(pi =>
pi.map(_ + 1).run(executor).get == Par.unit(2).run(executor).get)
```

Тест выразили, но он раздулся, как шар от спермы, замусорен деталями, и суть тонет в помоях — а они тут нахуй не упали. Заметьте, пацаны, дело не в выразительности API — да, мы можем сказать, что хотим, но без хелперов и с кривым синтаксисом intent заёбывается, как в старом Java-коде без Stream'ов.

ДОКАЗАТЕЛЬСТВО СВОЙСТВ Давай подкрутим. Первое наблюдение: `forAll` слишком общий для этого кейса, как универсальный нож-ложка-вилка из армии. Мы инпут не варьируем — просто хардкодный пример, чтоб не ебаться. Хардкод должен писаться так же с руками, как в классическом юнит-тесте. Закинем комбинатор для этого (на компаньоне `Prop`):

![Страница 0223, изображение 3](./assets/page-0223/image-03.png)

```scala
def verify(p: => Boolean): Prop
```
