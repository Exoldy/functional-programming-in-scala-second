#### ОТВЕТ 3.3

```scala
def setHead[A](as: List[A], hd: A): List[A] = as match
case Cons(_, tl) => Cons(hd, tl)
case Nil => sys.error("setHead of empty list")
```

Используем ту же хуйню, что и в определении `tail`: паттерн-матчинг на входном списке, разбираем каждый конструктор по косточкам.  
Как в `tail`, если напоролись на `Cons`, игнорим голову и биндим хвост на `tl`.  
Потом лепим свежий `Cons` с новой башкой `hd` и привязанным хвостом `tl`.  
А если `Nil` — валим ошибку.  
Или могли бы слить пустой список, но это уже на вкус и цвет.

#### ОТВЕТ 3.4

```scala
def drop[A](as: List[A], n: Int): List[A] =
if n <= 0 then as
else as match
case Cons(_, tl) => drop(tl, n - 1)
case Nil => Nil
```

Сначала чекнем, если просят дропнуть 0 или минус — возвращаем инпут как есть (`as`), без телодвижений.  
Иначе патчим список: если `Cons` — выкидываем голову и рекурсим `drop` на хвосте, отминусовывая единичку за слитую башку.  
Если же `Nil` — просто `Nil` и привет.  
Как в `tail` и `setHead`, могли бы орать ошибкой, если дроп больше длины, но мы выбрали по-другому — `drop` часто юзают, когда length вычисляют на лету.  
Если б фейлили, коллегам пришлось бы мерить размер заранее (типа `drop(as, n min length(as))`).  
Оба варианта ок, просто с практикой FP-либ набьёшь глаз на такие подвохи — я через это говно прошёл, знаю, где собака порылась.
