### 6.4.2 Nesting state actions

A pattern is beginning to emerge: we‚Äôre progressing toward implementations that don‚Äôt explicitly mention or pass along the `RNG` value. The `map` and `map2` functions allowed us to implement, in a rather succinct and elegant way, functions that were otherwise tedious and error-prone to write. But there are some functions we can‚Äôt write in terms of `map` and `map2`. One such function is `nonNegativeLessThan`, which generates an integer between 0 (inclusive) and `n` (exclusive):

```scala
def nonNegativeLessThan(n: Int): Rand[Int]
```

A first stab at an implementation might be to generate a non-negative integer modulo `n`:

```scala
def nonNegativeLessThan(n: Int): Rand[Int] =
map(nonNegativeInt)(_ % n)
```

This will certainly generate a number in the range, but it‚Äôll be skewed because `Int.MaxValue` may not be exactly divisible by `n`. So numbers less than the remainder of that division will come up more frequently. When `nonNegativeInt` generates numbers higher than the largest multiple of `n` that fits in a 32-bit integer, we should retry the generator and hope to get a smaller number. We might attempt this:

> Retry recursively if the Int we got is higher than the largest multiple of n that fits in a 32-bit Int.

```scala
def nonNegativeLessThan(n: Int): Rand[Int] =
map(nonNegativeInt): i =>
val mod = i % n
if i + (n-1) - mod >= 0 then mod else nonNegativeLessThan(n)(???)
```

This is moving in the right direction, but `nonNegativeLessThan(n)` has the wrong type. Remember that it should return a `Rand[Int]`, which is a function that expects an `RNG`. But we don‚Äôt have a `RNG` value to pass to the result of `nonNegativeLessThan(n)` there. What we would like is to chain things together so the `RNG` that‚Äôs returned by `nonNegativeInt` is passed along to the recursive call to `nonNegativeLessThan`.
